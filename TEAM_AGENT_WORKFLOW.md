# Командный процесс и агентный workflow

Этот файл нужен как рабочая инструкция для четырех программистов и их отдельных агентов.
Цель: работать в одном monorepo через Git без конфликтов по контрактам и без хаотичного дублирования логики.

## 1. Роли команды

### 1. Фронтенд

Отвечает за:
- `frontend/app`
- `frontend/admin`
- UI-слой, навигацию, экраны, локальный state.
- Отображение `entitlements snapshot` и сценариев unlock.

Что не трогает без согласования:
- `proto/`
- `contracts/`
- `api/openapi/`
- бизнес-логику денег, XP, долг, бан.

### 2. Python

Отвечает за:
- `services/python/interrogation-sim`
- `services/python/recommendation`
- `services/python/search`
- `services/python/analytics`
- `services/python/notification`

Что не трогает без согласования:
- Gateway.
- Go-контракты.
- C++-ядра.
- Любую бухгалтерию wallet-ledger.

### 3. C++ / C

Отвечает за:
- `services/cpp/slot-engine`
- `services/cpp/eta-engine`

Что не трогает без согласования:
- API gateway.
- UI.
- Транзакционные сервисы на Go.

### 4. Golang

Отвечает за:
- `gateway/bff-go`
- `services/go/platform/*`
- `services/go/product/*`
- `proto/`
- `api/openapi/`
- `contracts/`

Это владелец интеграции: он фиксирует контракты, поднимает каркас и связывает остальные языки.

## 2. Правило владения файлами

Один файл — один владелец.

Если файл принадлежит другому стеку, его не редактируют напрямую.
Изменения идут через:
- отдельную задачу,
- короткий PR,
- пересборку контрактов,
- согласование в `docs/`.

## 3. Как работать через Git

### Ветки

Рекомендуемый формат:
- `feat/frontend/<topic>`
- `feat/python/<topic>`
- `feat/cpp/<topic>`
- `feat/go/<topic>`

### Pull Request правила

- Один PR — одна тема.
- В PR не смешивать фронт, Python, C++ и Go, если это не контрактный PR.
- Контрактные PR должны быть самыми маленькими и идти раньше реализации.
- Если меняется `proto/`, все команды обязаны подтянуть изменения перед своей работой.

## 4. Агентная схема

У каждого программиста свой агент и свой контекст.

### Frontend-agent

Использует только:
- `frontend/`
- `api/openapi/`
- `docs/` для чтения контрактов.

Правило: не придумывать новые поля в ответах. Если поля не хватает, заявка уходит в контрактный PR.

### Python-agent

Использует только:
- `services/python/`
- `contracts/` как источник событий.

Правило: не дублировать логику money/xp, а только потреблять готовые события и писать свою доменную логику.

### C++-agent

Использует только:
- `services/cpp/`
- минимальные proto-контракты, если это нужно для интеграции.

Правило: C++ ядра считают только свою математику и не знают про UI.

### Go-agent

Использует:
- `gateway/`
- `services/go/`
- `proto/`
- `api/openapi/`
- `contracts/`

Правило: сначала закрепить контракт, потом писать реализацию.

## 5. Порядок совместной работы

1. Go-команда фиксирует контракты и каркас.
2. Фронтенд читает эти контракты и строит экраны.
3. Python закрывает AI/поиск/аналитику/уведомления.
4. C++ закрывает RNG и ETA-ядра.
5. Go собирает интеграцию, scheduler, wallet, entitlements и gateway.

## 6. Что считается конфликтом

Конфликтом считается ситуация, когда:
- два человека меняют один и тот же контракт без координации;
- один стек начинает описывать чужую бизнес-логику;
- фронтенд хранит скрытые правила unlock, которых нет в backend snapshot;
- wallet или progression меняются в нескольких местах.

## 7. Базовое правило синхронизации

Если нужен новый флаг, поле или событие:
1. Сначала правится договор в `docs/`.
2. Потом обновляется `proto/` или `contracts/`.
3. Потом подключаются реализации.
4. Только после этого фронт берет новое поле в UI.

## 8. Рабочий режим по дням

Каждый день команда делает одно и то же:
- синхронизация по изменениям,
- проверка контрактов,
- реализация только в своей зоне,
- короткий merge в main,
- повторная генерация/сборка при изменении контрактов.

## 9. Короткий чеклист перед коммитом

- Я не трогал чужую папку.
- Я не менял контракт без согласования.
- Я не добавил скрытую бизнес-логику в UI.
- Я не продублировал ledger или entitlement rules.
- Я обновил docs, если поменял поведение.

## 10. Итоговый принцип

Сначала контракт. Потом каркас. Потом реализация. Потом интеграция.
Так команда может работать параллельно и не ломать друг другу код.
