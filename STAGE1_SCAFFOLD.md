# Stage 1: первый каркас репозитория

Этот файл фиксирует стартовую структуру проекта и то, что должно появиться в первой итерации.

## 1. Что должно существовать в Stage 1

- Документация по архитектуре.
- Документация по ролям команды.
- Каркас API-контрактов.
- Каркас protobuf-контрактов.
- Каркас event contracts.
- Пустые точки входа для всех стеков.
- Папки, которые сразу отражают будущую структуру monorepo.

## 2. Итоговая структура верхнего уровня

```text
kids_of_gpt/
  README.md
  READMEPromts.md
  CODEGEN_MASTER_PLAN.md
  TEAM_AGENT_WORKFLOW.md
  STAGE1_SCAFFOLD.md

```

## 3. Правила stage 1

- Пока нет бизнес-логики, есть только структура и контракты.
- Пока нет контрактов, не писать реализацию, которая их обходит.
- Пока не выбран владелец папки, никто не меняет ее содержимое.
- `docs/` обновляется раньше, чем код.

## 4. Что уже считается готовым

- Общая карта сервисов зафиксирована.
- Роли четырех программистов описаны.
- Первый набор proto и event JSON лежит в репозитории.
- Есть placeholder-стабы для фронта, Go, Python и C++.

## 5. Что делаем после Stage 1

1. Закрепляем `registry-auth`.
2. Закрепляем `wallet-ledger`.
3. Закрепляем `progression-entitlements`.
4. Поднимаем первый happy path: регистрация -> логин -> Шарага -> XP -> деньги.
5. После этого начинаем дописывать продуктовые сервисы.

## 6. Как читать этот каркас

- `TEAM_AGENT_WORKFLOW.md` показывает, как не конфликтовать.
- `CODEGEN_MASTER_PLAN.md` показывает архитектуру.
- `docs/` показывает краткую рабочую спецификацию.
- `proto/`, `api/`, `contracts/` показывают границы данных.

## 7. Definition of Done для первого этапа

Stage 1 завершен, если:
- все папки созданы,
- у каждой зоны есть свой стартовый файл,
- правила работы команды записаны,
- следующий шаг можно брать без устных объяснений.
